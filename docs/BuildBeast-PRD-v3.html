1. Document purpose

This PRD defines the implementation-ready Milestone 1 for BuildBeast, the productized future of Command Center.

BuildBeast is the control plane for agent-assisted delivery: one system to intake work, route it to the right lane, coordinate execution, request approvals, surface meaningful signals, and preserve an audit trail.

  • rigorous domain definitions
  • a real Milestone 1 scope table
  • concrete routing rules
  • concrete approval spec
  • concrete signal spec
  • concrete document/artifact spec
  • core user flows
  • measurable acceptance criteria
  • a separate concise 1-page executive brief

2. Product thesis

Teams do not just need AI that can do work. They need a trustworthy operating layer between messy requests and shipped outcomes.

Messy request → structured intake → explainable routing → supervised execution → approvals → readable signals → auditable outcome

Product promise: Give operators one place to intake, route, orchestrate, supervise, and audit agent-assisted work until it ships.

Milestone 1 objective: Use Command Center V1 as the proving ground for the core BuildBeast abstractions and make them credible enough that the team can build against this document without re-litigating the model every sprint.

3. Problem statement

  1. Requests arrive vague — users describe goals in chat, not structured fields.
  2. Routing is inconsistent — ownership depends on who saw the request first.
  3. State is fragmented — projects, tasks, jobs, approvals, docs, events, and usage live in different places.
  4. Approvals are under-specified — decisions arrive without enough context.
  5. Signals are noisy — raw event feeds are not good operational summaries.
  6. Documents are weakly integrated into execution — PRDs and uploaded docs exist, but are not clearly defined as part of the operating model.

4. Product goals and non-goals

Goals

  1. Make project creation fast and structured.
  2. Recommend owner and QC lanes immediately and explain why.
  3. Give every project a single trustworthy workspace.
  4. Distinguish durable work items from execution attempts.
  5. Make approvals explicit, contextual, and auditable.
  6. Convert raw events into readable, high-signal updates.
  7. Tie documents and usage back to projects.

Non-goals for Milestone 1

  1. Multi-tenant SaaS
  2. Generic workflow builder
  3. Full enterprise permissions model
  4. Runtime-agnostic adapters beyond OpenClaw
  5. Billing and self-serve onboarding
  6. Replacing Linear/Jira/Asana for every workflow

5. Target users

Primary user now: Operator / chief of staff / delivery lead running cross-functional, agent-assisted work.

Needs: fast triage, owner/QC recommendation, intervention queue, readable progress, confidence in the audit trail.

Secondary users: requester, owner lane, QC approver, contributor, stakeholder.

Best future ICPs: AI-native agencies / studios, founder-led product teams, internal ops / innovation teams.

6. Locked domain definitions

Project: the durable business container for a desired outcome.

Intake: the structured request payload stored on the project and used for routing.

Team / lane: a functional ownership queue such as Product, Design, Engineering, Marketing, QA.

Task: a durable scoped work item within a project. Milestone 1 should treat sprint_items as tasks/work items even if the table name remains unchanged.

Job / run: a discrete execution attempt or assignment. A job is not the same thing as a task.

Approval: a first-class decision object requiring human review.

Event: a durable record of a meaningful change.

Signal: a human-readable projection derived from events and state changes.

Document / artifact: a file or link attached to project context.

PRD: a structured planning document attached to a project.

7. Product principles

  1. Human override is first-class.
  2. Routing is product value, not setup glue.
  3. Project, task, and job must remain distinct.
  4. Approvals are first-class objects, not status flags.
  5. Signals should increase trust, not noise.
  6. Documents belong inside the operating model, not beside it.
  7. Realtime is a projection; durable state is the truth.
  8. Internal Vertillo personas are presets, not the long-term product model.

9. Functional specification

9.1 Intake

Turn an ambiguous request into structured data that can drive routing and execution.

Required fields: project name, shape, stage, confidence.

Optional fields: context, capabilities, goals/notes, uploaded docs.

Acceptance criteria: hybrid/uncertain requests have a safe path, intake persists to the project record, and the summary is readable in list/detail views.

9.2 Routing

Recommend a primary owner lane and QC lane immediately using deterministic, explainable logic.

  • Product owns first when confidence is low, stage is early, or request is hybrid/strategy-heavy.
  • Marketing owns first for launch/growth-heavy work.
  • Engineering owns first for build-heavy work or new-product shape.
  • Design owns first when UX/UI is the main need and the request is already scoped.

QC rules: Engineering → QA, Design → Product, Marketing → Product, Product → QA.

9.3 Project creation

  1. Create projects row
  2. Persist type, team_id, description, intake, intake_summary
  3. Set status = active and progress_pct = 0
  4. Create starter tasks per routed team using sprint_items
  5. Upload/store any project docs in project_documents
  6. Sync/roll up project state

9.4 Tasks and jobs

Tasks are durable project-scoped work. Jobs are execution attempts against that work. They must remain distinct.

9.5 Approvals

Approvals are first-class decisions with context, severity, status, and audit trail. Owner lane cannot self-QC for the same gate.

9.6 Signals

Signals are curated, stakeholder-readable updates derived from events and state changes. Minimum canonical signal events include project created/routed, task created/blocked/completed, approval requested/decided, document uploaded, PRD created/updated, and project completed.

9.7 Documents and PRDs

Documents are operational context, not detached attachments. Project detail must show docs in one predictable section, and uploads should generate signals.

10. Core user flows

  1. Create a project from ambiguous request — structured intake, routing preview, project creation, starter tasks, workspace opens.
  2. Review routing on a hybrid request — Product owns first, supporting teams are pulled in, rationale is visible.
  3. Run delivery and monitor progress — starter tasks, state updates, progress and signals refresh, blocked work is visible.
  4. Request and resolve approval — queue, context, decision trail, project updates.
  5. Inspect project documents and planning context — intake, goals, routing, docs, PRDs, signals, and tasks all live together.

11. Information architecture

Top-level modules: Dashboard, Projects list, Project workspace, Approvals queue, Agents/runtime registry, Usage view.

Required project workspace sections: Overview/brief, Tasks/work board, Recent signals, Documents, Teams, Approvals summary.

12. Data model grounding

Current tables directly relevant to M1: projects, teams, team_members, sprint_items, jobs, approvals, prds, project_documents, agent_events, ai_usage_events, usage_rollup_minute.

13. Success metrics

  • Median time to create project: < 2 minutes
  • Routing stickiness: ≥ 80%
  • Approval clarity: ≥ 90%
  • Project state trust: ≥ 90%
  • Signal usefulness: operators prefer signals to raw logs for daily triage
  • Document visibility: docs accessible from workspace for ≥ 95% of projects with uploads

14. Risks

  1. Overfitting to Vertillo internal language
  2. Confusing tasks and jobs
  3. Treating raw events as signals
  4. Weak approval state model
  5. Documents behaving like detached attachments
  6. Trying to jump to workspaces before M1 is trusted

15. Recommended implementation sequence

  1. Lock terminology in product copy
  2. Finish project workspace as the single operational page
  3. Strengthen approvals queue + project approval context
  4. Formalize signal generation rules from current events/state
  5. Tighten document + PRD display and linking
  6. Validate routing explanation and override path

16. Final summary

BuildBeast should not be treated as just a dashboard or OpenClaw with UI.

Milestone 1 is where the product model becomes concrete: structured intake, explainable routing, durable projects, tasks distinct from jobs, explicit approvals, readable signals, and documents inside the operating loop.

If M1 is done correctly, Command Center becomes a credible internal control plane and the foundation for productization.
