1. Product Summary

BuildBeast is the future product direction emerging from Command Center V1.

Command Center V1 is currently the internal proving ground: a control interface used by Vertillo to intake projects, route work to specialist teams, track execution, manage approvals, and surface operational signals.

BuildBeast is the productized vision of that system.

BuildBeast will begin as a control plane that can orchestrate an existing OpenClaw environment and, over time, evolve into a runtime-capable system that can operate independently without requiring an OpenClaw gateway.

At its core, BuildBeast helps users turn ideas into shipped outcomes through structured intake, intelligent routing, AI-assisted execution, human approvals, and operational visibility.

2. Vision

Long-term vision

BuildBeast becomes the operating system for turning ideas into executed projects.

  • describe what they want to build
  • provide context, goals, and supporting documents
  • have the system scope and route the work intelligently
  • coordinate design, product, engineering, marketing, and QA work
  • approve key decisions at the right moments
  • monitor progress, blockers, costs, and outcomes
Product framing: BuildBeast is the control plane for agent operations.

3. Problem

People have ideas they want executed, but the path from idea to shipped outcome is fragmented.

  • intake is vague and poorly structured
  • project routing depends on whoever reads the request
  • task assignment is inconsistent
  • project status is noisy or misleading
  • approvals happen too late or in the wrong place
  • AI tools can perform work but lack a reliable operational control plane
  • internal tools often overfit to one operator's workflow and do not productize well

4. Opportunity

BuildBeast can become the layer that structures intake, maps user needs to execution lanes, coordinates project work, surfaces decisions and blockers, provides trustworthy operational visibility, and creates a repeatable delivery model for humans and AI working together.

5. Product Principles

  1. Human override is first-class
  2. Routing is a product capability
  3. Project != run
  4. The event log is sacred
  5. Opinionated defaults beat empty flexibility
  6. Governance and visibility matter
  7. Abstractions should survive productization

6. Command Center V1 vs BuildBeast

Command Center V1

Internal proving ground for Vertillo. Used to validate intake, routing, project pages, signals, approvals, and team assignment.

BuildBeast

Future product that exposes the best abstractions from Command Center V1 and generalizes them for broader users.

7. Core Jobs to Be Done

Primary job: “When I have something I want built, help me turn it into structured work, assign it to the right lanes, and keep it moving until it ships.”

8. V1 Scope (Internal)

  • guided project intake
  • structured project creation
  • project-level task orchestration
  • team/owner assignment
  • recent signals / activity summary
  • approvals / needs-attention queue
  • project visibility on desktop and mobile
  • supporting document upload
  • QC-aware operating model

9. Product Requirements

9.1 Guided Intake

Users should create projects through a structured, selection-first flow with plain language, examples, hybrid support, and optional notes/documents.

9.2 Routing

The system should route work using structured intake signals, not a single rigid type field.

9.3 Project Workspace

Each project should answer what this project is, current status, who owns what, what is blocked, what changed, and what needs approval.

9.4 Recent Signals

Activity should be curated, human-readable, and high-signal.

9.5 Supporting Documents

Projects should support attached briefs, PRDs, screenshots, mockups, notes, and technical context.

9.6 Approvals / Needs Attention

The system should clearly surface approvals, blocked work, stalled projects, and intervention points.

9.7 Runtime Orchestration

In the first phase, BuildBeast orchestrates through the current OpenClaw environment.

9.8 Future Runtime Independence

Over time, BuildBeast should support execution without depending entirely on an OpenClaw gateway.

10. Architecture Direction

  1. System of record — projects, tasks, approvals, teams, documents, events, usage
  2. Orchestration / control layer — routing, task state, approvals, triggers, signals
  3. Runtime adapter layer — OpenClaw first, other adapters later

11. Risks

  • Overfitting to Milo/Vertillo workflow
  • Confusing internal dashboard behavior with product-ready abstractions
  • Inconsistent domain models
  • Noisy or untrustworthy status surfaces
  • OpenClaw-specific implementation details becoming permanent concepts
  • Trying SaaS before identity, tenancy, security, and workspace abstractions exist

12. Non-Goals (for now)

  • launching public SaaS immediately
  • solving every PM workflow
  • replacing every external tool at once
  • exposing raw runtime concepts directly to end users

13. Success Criteria

Internal V1 success

  • BuildBeast is tracked as a real project inside Command Center V1
  • intake feels guided and useful
  • routing feels smarter and less rigid
  • project pages are trusted operational views
  • approvals and recent signals are genuinely useful

Productization success

  • the model can be generalized beyond Milo-specific assumptions
  • core abstractions survive contact with outside users/teams
  • routing and intake remain understandable without coaching

14. Immediate Next Steps

  2. Attach this PRD as the supporting project document
  3. Use BuildBeast to test intake, routing, project page usefulness, approvals, recent signals, and team assignment

15. Suggested First Milestone

BuildBeast Foundation

Make Command Center V1 good enough to manage BuildBeast credibly as a real internal product.

  • stable project intake
  • clean project page experience
  • useful team/owner visibility
  • approvals that work
  • recent signals that matter
  • supporting document attachment flow
  • reliable task/project state updates

16. Final Summary

Command Center V1 is the proving ground. BuildBeast is the product.

The job now is not to pretend the internal tool is already a SaaS. The job is to use it to discover and solidify the abstractions that can become one.
